# LSB (경락이) 구축 계획서 — 실행판 v1.1

> 경영지원실 Slack 지식 에이전트 `@경락이`. 인계요약(v0.1) 기반, 확정된 방향에 맞춰 P0 빈칸을 채우고 P1~P6 실행 체크리스트로 정리한 문서.

## 네이밍 (확정)

- **한글명**: `@경락이`
- **영어명**: **LSB (Loadcomplete Support Bot)** ← 변경 (기존 LCB / LoadComplete Context Bot)
- **소속·용도**: 경영지원실 봇. 보고 내용 검색 + 그 결정의 맥락까지 아는 에이전트.
- **오너/구축**: 박성훈 (관리자 권한 보유, 봇 구축 경험 있음 → 직접 구축 진행)

## 확정된 방향 (이번 세션 결정)

- **데이터 소스**: ① Slack 메시지·스레드 ② Slack에 올린 첨부파일(PDF/한글/엑셀 등) ③ 외부 저장소(Google Drive / Notion)
- **이번 결과물**: 실행 가능한 구축 계획서 (본 문서)
- **최종 봇 형태**: **진짜 Slack 앱** — 팀원이 채널에서 `@경락이`를 멘션하면 답하는 봇 → 별도 호스팅(상주 서버) 필요
- **용도**: 지나간 보고문서가 어디 있는지 + 그 결정의 맥락(언제·왜·누가·이후 변화)까지 출처와 함께 설명. 더불어 일반 대화 가능한 에이전트.

## 1. 최종 그림 (데이터 흐름)

```
[Slack 채널 메시지/스레드] ─┐
[Slack 첨부파일]          ─┼─▶ 수집 ─▶ 정규화·청킹 ─▶ 임베딩 ─▶ 벡터DB(+메타데이터: 채널/작성자/시간/출처/권한)
[Google Drive / Notion]   ─┘                                          │
                                                                       ▼
팀원: "@경락이 작년 보안감사 보고서 어디 있어?"                      검색(RAG)
        │                                                  의미+키워드 하이브리드 / 권한필터 / 최신성 가중
        ▼                                                              │
   Slack 앱(Socket Mode) ──── app_mention 이벤트 ────▶  LLM 답변 생성 ─┘
        ▲                                              근거 인용 + 출처 링크 + 맥락(타임라인)
        └──────────────  chat:write 로 답변 전송  ◀───────────────────┘
```

핵심 차별점은 4번 기둥 **맥락 추적**: 단순 검색이 아니라 "이 결정이 언제·왜·누가, 이후 어떻게 바뀌었는지"를 시간축으로 연결.

## 2. P0 결정 — 권고안 (빈칸 채움)

| 항목 | 인계 시점 | 권고안 (이번 계획) | 비고 / 결정 필요 |
|---|---|---|---|
| 권한 모델 | B 권고 | **B (화이트리스트)로 시작, 색인에 권한 메타데이터 필수 부착** | 나중에 A로 졸업 가능하도록 1단계부터 메타데이터 심기 |
| 대상 채널(1개) | 미정 | **민감도 낮은 P&C(피플&컬처) 단일 채널로 dogfooding** | 박성훈이 채널 1개 지정 — 인사/급여 raw 데이터 없는 곳 |
| 외부 전송 정책 | 미정 | **P1(읽기·내부확인)까지는 외부전송 0 → 안전. P3 답변생성부터 LLM 전송 정책 확정 필요** | 보안/경묵 확정. "어떤 LLM에/어디까지" 명문화 |
| 기술 스택 | 미정 | **Python + Slack Bolt(Socket Mode) + 벡터DB + 임베딩/LLM API** (아래 §3) | P1은 읽기 스크립트만 |
| 개발 담당 | 미정 (박성훈 vs 성훈 정) | **확정 필요** — 비개발자 단독이면 P1~P2까지는 Cowork 보조로 가능, P3+는 구현자 권장 | 폴백: 경묵 |
| 외부 소스 범위 | (신규) | **P1은 Slack만. Drive/Notion은 P2 이후 커넥터로 추가** | Drive·Notion 커넥터 연결 권한 확인 |

> ⚠️ **착수 전 반드시 닫아야 할 관문 2개**: ① Slack 앱 설치(관리자 승인) ② 외부전송 허용 여부. 둘 중 하나라도 막히면 단계 조정 필요.

## 3. 기술 스택 권고

| 레이어 | 권고 | 이유 |
|---|---|---|
| Slack 연동 | **Slack Bolt for Python + Socket Mode** | 공개 HTTP 엔드포인트/서버 노출 불필요 → 사내·온프레미스 봇에 최적. WebSocket으로 이벤트 수신 |
| 봇 실행 환경 | 사내 상시 가동 머신 또는 소형 클라우드 인스턴스 1대 | Socket Mode는 상주 프로세스 필요(멘션 대기) |
| 수집/파싱 | Slack Web API(`conversations.history`, `conversations.replies`, `files`) + 첨부 파서(PDF/docx/xlsx → 텍스트) | 메시지·스레드·파일 본문 추출 |
| 임베딩 + 벡터DB | 임베딩 API + 경량 벡터DB(예: 로컬 Chroma/pgvector 등) | 1채널 규모는 경량으로 충분 |
| 답변 LLM | LLM API (외부전송 정책 확정 후 선택) | 근거 기반 답 + 출처 인용 |
| 외부 소스 | Google Drive / Notion 커넥터(MCP) 또는 공식 API | 보고서가 외부 저장 시 본문/메타데이터 수집 |

> 스택은 권고일 뿐 — 개발 담당 확정 후 그 사람이 익숙한 도구로 조정 가능. 핵심 제약은 "Socket Mode 상주 + 권한 메타데이터 색인" 둘뿐.

## 4. Slack 앱(@경락이) 구성 체크리스트

진짜 멘션 응답 봇이므로 아래가 필수.

**앱 생성 & 모드**
- [ ] api.slack.com에서 앱(LSB / @경락이) 생성 — 관리자 권한 보유로 직접 승인 가능
- [ ] **Socket Mode 활성화** → App-Level Token(`xapp-…`) 발급 (`connections:write`)
- [ ] Bot Token(`xoxb-…`) 발급

**Event Subscriptions (멘션 응답용)**
- [ ] `app_mention` 이벤트 구독 → 팀원이 `@경락이` 부를 때 트리거

**Bot Token Scopes**
- [ ] `app_mentions:read` — 멘션 수신
- [ ] `chat:write` — 답변 전송
- [ ] 비공개 채널 읽기: `groups:history`, `groups:read`
- [ ] 공개 채널 읽기: `channels:history`, `channels:read`
- [ ] `users:read` — 작성자 이름 해석
- [ ] `files:read` — 첨부파일 본문 수집
- [ ] (검색 보조) `search:read` 검토 — 사용자 토큰 필요 시 별도 고려

**설치 & 초대**
- [ ] 워크스페이스에 앱 설치(관리자 승인)
- [ ] 대상 채널에서 `/invite @경락이`

## 5. 단계별 실행 계획 (P1 → P6)

### P1 — 채널 읽기 ← 현재
- **목표**: 지정 P&C 채널의 메시지·스레드·첨부 목록을 읽어와 표시. (AI 요약·외부전송 없음 = 순수 읽기, 안전)
- **할 일**
  - [ ] (박성훈) 대상 채널 1개 선정 + 앱 설치 승인 + 봇 초대
  - [ ] (구현자) `conversations.history`/`replies`로 메시지·스레드 수집 스크립트
  - [ ] (구현자) `files.list`/파일 본문으로 첨부 메타데이터·텍스트 추출
  - [ ] 증분 동기화 토대(마지막 수집 시점 이후만 가져오기) 설계
- **완료 기준**: 채널 N일치 메시지 + 첨부 목록이 로컬에 안전히 떨어지고, 다시 돌리면 새 것만 추가됨
- **관문**: 앱 설치 승인 ②(외부전송은 P1에선 불필요)

### P2 — 색인 + 검색
- **목표**: 수집물을 청킹·임베딩하여 벡터DB에 적재, 의미+키워드 하이브리드 검색.
- **할 일**
  - [ ] 청킹 규칙(스레드 단위/메시지 묶음) 정의
  - [ ] 메타데이터 스키마: 채널 / 작성자 / 시간 / 출처링크 / **권한태그** 부착 (← A 졸업 대비 필수)
  - [ ] 임베딩 적재 + 검색 API
  - [ ] **외부 소스 1차 연결**: Google Drive / Notion 커넥터로 보고서 본문·메타 수집해 같은 색인에 합치기
- **완료 기준**: "보안감사 보고서" 같은 자연어로 검색 시 관련 메시지·파일이 출처와 함께 반환

### P3 — 근거형 Q&A
- **목표**: 검색 결과를 근거로 LLM이 답 + **출처 메시지/파일 링크** 첨부. 모르면 "모른다".
- **할 일**
  - [ ] **외부전송 정책 확정**(보안/경묵) — 이 단계의 관문
  - [ ] 프롬프트: 근거 인용 강제, 추측 금지
  - [ ] Slack 앱에 `app_mention` 핸들러 연결 → 멘션 시 검색→답변 파이프라인
- **완료 기준**: 채널에서 `@경락이 작년 X 결정 어디 있어?` → 출처 링크 달린 답변 반환

### P4 — 맥락 추적 ⭐ (차별점)
- **목표**: 시간축으로 논의·결정 전개 연결 ("언제 왜 누가, 이후 어떻게 바뀜").
- **할 일**
  - [ ] 결정/변경 이벤트 추출·연결(타임라인 구성)
  - [ ] 답변에 "이후 N월 이 결정이 ○○로 수정됨" 식 맥락 추가
- **완료 기준**: 단발 답이 아니라 결정의 변천사를 짚어줌

### P5 — 권한·확장
- **목표**: 권한 모델 A(멤버십 추적), 다채널, 외부전송 정책 전면 적용.
- **할 일**
  - [ ] P2에서 심은 권한 메타데이터로 사용자별 답 필터
  - [ ] 채널 확장 + 민감 채널 편입 기준
- **완료 기준**: 사용자가 볼 권한 있는 내용만 답에 노출

### P6 — 평가·운영
- **목표**: 정확도 루프, 비용·지연·안정성 모니터링.
- **할 일**
  - [ ] 근거성 평가 + 👍/👎 피드백 수집
  - [ ] 비용·지연·가동 모니터링, 오답 회귀 점검
- **완료 기준**: 정확도·비용·안정성이 측정·개선되는 운영 루프 가동

## 6. 외부 소스(Drive / Notion) 연동 계획

- **시점**: P2(색인) 이후 추가 — P1은 Slack만으로 토대 검증.
- **방식**: Drive/Notion 커넥터(MCP)로 보고서 본문·메타데이터(제목/작성자/수정시각/링크) 수집 → Slack 데이터와 **동일 색인·동일 메타스키마**로 합침.
- **출처 표기**: 답변에서 "Slack 메시지 vs Drive 문서 vs Notion 페이지"를 출처 타입으로 구분해 링크.
- **권한 주의**: 외부 문서도 권한태그 부착 — 민감 보고서가 화이트리스트 밖 사용자에게 노출되지 않게.

## 7. 보안·권한 설계 (성패 요인)

- 경영지원실 = 인사·급여·계약 등 민감정보 → **기능 품질보다 정보경계 설계가 성패를 가름**.
- **1단계 B(화이트리스트)**: 신뢰 사용자 명단만 봇 사용, 봇은 고정 채널만 읽음 → 단순·빠름.
- **색인 단계부터 권한 메타데이터 필수**: B로 시작해도 A(멤버십 추적)로 졸업하려면 처음부터 권한태그가 있어야 함.
- **외부전송 정책**: 채널/문서 내용을 외부 LLM에 보내도 되는지, 답을 어디까지 내보내는지 명문화 필요(보안/경묵). 순수 읽기·내부 색인(P1)은 외부전송 없이 가능하므로 정책 확정 전에도 안전.
- **PII·감사 로그**: 누가 무엇을 물었고 무엇이 노출됐는지 로깅.
